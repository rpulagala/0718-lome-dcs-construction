# Claude Code Prompt: Build a Construction Work Intake and Tracking Web Application

You are a senior full-stack engineer, product architect, UX designer, database architect, security engineer, and QA automation engineer.

Your task is to design and build a production-ready MVP for a construction company called **DCS Construction**.

The application will allow prospective customers to submit construction work requests and allow DCS employees and administrators to review, assign, schedule, track, and manage those requests through completion.

Use the attached reference document as the original business and workflow specification:

`DCS Construction Site Map 3 (1).pdf`

Read all pages of the PDF before designing or coding. Use the document as a source of requirements, but improve the workflows and user experience where appropriate. Do not blindly reproduce the old mobile mockups. Build a modern, responsive web application that works well on phones, tablets, and desktop browsers.

## 1. Primary Product Goal

Build a construction project intake and tracking application with three main user groups:

1. Public customer or prospective customer
2. DCS employee
3. DCS principal administrator

The MVP must support the complete lifecycle:

```text
Customer submits request
→ Confirmation is sent
→ Request appears in employee dashboard
→ Employee reviews and qualifies request
→ Site visit is scheduled
→ Request is assigned
→ Estimate and project information are tracked
→ Status is updated
→ Customer receives appropriate notifications
→ Request is completed, cancelled, or archived
```

The application should initially serve one construction company, DCS Construction. Structure the code and database cleanly enough that multi-company support could be added later, but do not build multi-tenancy in the MVP unless it is needed for sound architecture.

## 4.4 Submission Processing

When the customer submits a request:

1. Validate the form on both client and server.
2. Use a database transaction where appropriate.
3. Create a unique, human-readable request number, such as:

```text
DCS-2026-000123
```

4. Save the request and image records.
5. Add an initial status-history record.
6. Record a timeline activity for the submission.
7. Send a confirmation email to the customer.
8. Notify designated DCS employees or a shared intake mailbox.
9. Show a confirmation page.

Confirmation page message:

> Thank you for your interest in DCS Construction. Your request has been received. Our team will review it and contact you within 48 business hours.

Display:

* Request number
* Submission date
* Summary of submitted information
* Expected response time
* DCS contact information

Prevent accidental duplicate submissions caused by double-clicking or browser retries. Use an idempotency mechanism.

```text
NEW
REVIEWING

ESTIMATE_IN_PROGRESS

APPROVED
PROJECT_SCHEDULED
IN_PROGRESS
ON_HOLD
COMPLETED

```

Also create a simpler customer-facing status mapping, for example:

```text
Request Received
Under Review
Site Visit Scheduled
Estimate Being Prepared
Estimate Sent
Project Scheduled
Work in Progress
Completed
Closed
```

Requirements:

* Record every status change.
* Store who made the change.
* Store the timestamp.
* Allow an optional reason or note.
* Prevent invalid transitions where appropriate.
* Trigger selected notifications from meaningful status changes.
* Do not expose every internal status to customers.

# 14. Suggested Database Model

Design a normalized relational schema.

Likely entities:

```text
User
Role
Customer
Address
WorkRequest
ProjectCategory
WorkRequestPhoto
WorkRequestAttachment
WorkRequestStatusHistory
WorkRequestAssignmentHistory
WorkRequestNote
WorkRequestActivity
Task
SiteVisit
SiteVisitHistory
Communication
Estimate
Project
ProjectMilestone
Notification
EmailLog
CustomerAccessToken
CompanySetting
AuditLog
```

Important modeling requirements:

* Use UUIDs or secure non-sequential internal IDs.
* Use a separate human-readable request number.
* Include `createdAt` and `updatedAt`.
* Use soft deletion or archival for important business records.
* Preserve historical assignments and statuses.
* Create appropriate indexes.
* Avoid putting all information into large JSON columns.
* JSON may be used for non-critical metadata but not as a substitute for a proper schema.
* Store timestamps consistently in UTC.
* Display dates in the configured company timezone.
* Avoid permanently deleting customer submissions through normal UI actions.

Create a Prisma schema and include an entity-relationship diagram in Mermaid syntax in `docs/DATA_MODEL.md`.

* A public customer can submit a construction request with photos.

* The customer sees a confirmation and receives an email.
* Internal users are notified.
* Employees can see and search incoming requests.

* The application is responsive and accessible.
* Local setup and deployment steps are documented.
